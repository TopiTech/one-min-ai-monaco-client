# 1min.ai Monaco Client — 包括的コードレビュー

## 概要
1min.ai APIをプロキシするExpressサーバー + Monaco EditorベースのブラウザAIクライアント。
チャット、画像生成/編集、コード生成/補完、ファイルシステム操作、コマンド実行を提供。

---

## 1. セキュリティ

### 1.1 APIキー漏洩 [重大]
- `.env` に実際のAPIキーがコミットされている（`ff66e197...`）。`.gitignore`は`.env`を除外しているが、リポジトリに既に含まれている可能性がある。
- **修正**: `.env`の内容を即座にローテーションし、git履歴から削除を検討。

### 1.2 BFF認証の堅牢性
- `localBffAuth()` は二要素検証（HttpOnly cookie + x-local-bff-token header）+ same-origin検証を実装。設計は良好。
- `timingSafeEqual` を使用したトークン比較は正しい。
- **軽微**: `data-bff-token`属性がbodyに注入されるが、DOMPurifyでサニタイズされていない（`escapeHtmlAttr`で対応済み）。OK。

### 1.3 CSP（Content Security Policy）
- `unsafe-inline` がscriptSrcに含まれる（Monaco AMDローダーのため必要）。コメントで言及済み。
- **推奨**: 本番環境ではnonceベースに移行を検討。

### 1.4 コマンド実行 [重大]
- `ENABLE_COMMAND_EXECUTION=true` + `ALLOWED_ROOTS=C:\Users\mibu0\Documents\develop` が設定されている。
- `DANGEROUS_PATTERNS`によるブロック + コマンドホワイトリスト + シェルインジェクション防止（`SHELL_INJECTION_PATTERN`）は適切。
- `checkCommandSafety()` は二重にチェック（コマンド実行時 + 承認時）。
- **軽微**: `ALLOWED_COMMAND_NAMES`に`node`が含まれているが、`node -e`は`DANGEROUS_PATTERNS`でブロック。整合性はOK。

### 1.5 ファイルシステム操作
- `fs-guard.js`の`validatePath()`は realpath 解決 + allowed roots チェック + TOCTOU対策（`revalidateRealPath`）。
- `WRITE_PROTECTED_PATH_PREFIXES`でサーバーコード自体の書き込みを防止。
- **良好**: .env、.git、node_modules、server.js、routes/、config/、public/が保護対象。

## 2. API使用法の正確性（1min.aiドキュメント照合）

### 2.1 Chat with AI API
- `POST /api/chat-with-ai` を使用。`type: "UNIFY_CHAT_WITH_AI"` はドキュメント通り。
- リクエスト構造: `promptObject.prompt`, `promptObject.settings.webSearchSettings`, `promptObject.settings.historySettings` はAPI仕様と一致。
- `attachments` は `promptObject` 内にネスト（正しい）。
- **整合性OK**: APIドキュメントの推奨構造に準拠。

### 2.2 AI Feature API
- `POST /api/features` を使用（コード生成、画像生成、画像編集）。
- `type: "CODE_GENERATOR"`, `type: "IMAGE_GENERATOR"`, `type: "IMAGE_EDITOR"` はAPI仕様通り。
- **整合性OK**。

### 2.3 Asset API
- `POST /api/assets` でmultipart/form-dataアップロード。
- `normalizeAssetResponse()`が`asset.key`, `fileContent.path`, `asset.location`を吸収。構造はAPIレスポンスと一致。
- **整合性OK**。

### 2.4 API認証ヘッダー
- `API-KEY` と `Authorization: Bearer` の両方を送信（`api-client.js:108-111`）。
- ドキュメントでは `API-KEY` ヘッダーが推奨。両方送信は冗長だが害はなし。

## 3. エラーハンドリング

### 3.1 適切な点
- `callOneMin()` は429リトライ + 指数バックオフ + ジッター。
- 非等冪性（conversations, assets）ではリトライを無効化（`idempotent: false`）。
- ストリーミングではクライアント切断時にAbortControllerでリソース解放。
- グローバルエラーハンドラーが全エラーをキャッチ。

### 3.2 改善点
- `routes/ai.js:233-243`: ストリーミングエンドポイントで、upstreamが非ストリーミングレスポンスを返した場合のフォールバックがある。堅牢。
- `isFailedResponse()` は `aiRecord.status` が "SUCCESS" または "COMPLETED" でない場合を失敗と判定。1min.aiの仕様と一致。

## 4. コード品質

### 4.1 アーキテクチャ
- サーバーサイド: Express + ルーター分離（ai.js, fs.js, agent.js）+ 設定集中管理（config/）。
- クライアント: Vanilla JS + Monaco Editor + DOM操作。フレームワーク不使用。
- APIキーはサーバー側のみ。フロントエンドに露出なし。**BFFパターンとして正しい**。

### 4.2 重複コード
- `routes/ai.js`の`/chat`と`/chat/stream`でリクエストバリデーションが重複。抽出可能だが重大ではない。

### 4.3 フロントエンド
- `app.js`は1600行超。巨大だが、Monaco Editor統合、エージェント、チャット、画像生成を1ファイルに集約。
- `public/js/api.js`の`extractImages()`は1min.aiのレスポンス構造の変動を幅広く吸収。堅牢。
- DOMPurify + markedでMarkdownサニタイズ。XSS対策として適切。

### 4.4 テスト
- 11個のテストファイルが存在。`agent-routes`, `api-client`, `command-runner`, `fs-guard`, `routes`, `server`, `web-search` をカバー。
- Jest + supertest。`--experimental-vm-modules` でESM対応。

## 5. 1min.ai APIとの差分・問題点

### 5.1 レート制限
- サーバー設定: 180 req/min（デフォルト）。1min.aiドキュメントでも180 req/min。一致。
- `/api/chat` は300/min、`/api/code` は600/minに緩和。自動補完は高頻度なので適切。

### 5.2 モデル一覧
- `config/models.js`にハードコードされたモデルリスト + 30分ごとの動的取得（`/api/models`）。
- `fetchModels()` が404を「機能利用不可」として処理（`H-5`コメント）。現在の1min.ai APIでは`/api/models`が公開されていない可能性あり。フォールバックが機能する。

### 5.3 会話作成
- `POST /api/conversations` で `type: "UNIFY_CHAT_WITH_AI"` を使用。ドキュメントの推奨と一致。

## 6. 改善提案

### 優先度: 高
1. **`.env`のAPIキーローテーション**: リポジトリに実際のキーが含まれている。即座に無効化・再生成。

### 優先度: 中
2. **CSP nonce導入**: 本番環境で`unsafe-inline`を排除。
3. **ストリーミングのタイムアウト**: クライアント側で最大ストリーミング時間を制限（例: 5分）。
4. **モデルリストの動的取得**: `/api/models`が404の場合のフォールバックは機能しているが、定期的な確認ロギングを追加。

### 優先度: 低
5. **app.jsの分割**: チャット、画像、エージェント、Monaco関連を別モジュールに分割。
6. **APIクライアントのリファクタリング**: `routes/ai.js`のリクエストバリデーション重複を解消。
