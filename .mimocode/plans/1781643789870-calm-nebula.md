# Code Review Plan: one-min-ai-monaco-client

## Overview

BFF (Backend-For-Frontend) architectureで1min.ai APIをプロキシするMonaco Editorクライアント。セキュリティ対策が手厚く実装されている。以下、カテゴリ別に詳細レビューを実施。

---

## 1. 総評

全体的に**高品質なコードベース**。BFFパターンによるAPIキー隔離、CSRF対策（二重トークン+Same-Origin検証）、CSP設定、レート制限、ファイルシステムガード、コマンド実行のサンドボックス化など、セキュリティへの配慮が非常に深い。MVPとしては優秀。

---

## 2. Blocker / High（即時修正推奨）

### H-1: API-key ヘッダー二重送信（api-client.js:84-86）

```js
headers: {
  Authorization: `Bearer ${apiKey}`,
  "API-KEY": apiKey,
  ...headers,
}
```

1min.ai公式ドキュメントでは `API-KEY` ヘッダーのみ要求。`Authorization: Bearer` は未承認ヘッダー。二重送信は：
- APIキーの漏えいリスク増大（CORS/プロキシログに二重に記録）
- 一部プロキシ/CDNが `Authorization` ヘッダーをログに残す場合がある

**修正**: `Authorization: Bearer` を削除し、`API-KEY` のみ使用。もしくは1min.aiが両方必要ならドキュメントに記載を確認。

### H-2: ストリーミングchatで `response.clone()` の後のJSON解析失敗時の未処理パス（ai.js:217-229）

```js
if (response.headers.get("content-type")?.includes("application/json")) {
  const data = await response.clone().json().catch(() => null);
  if (isFailedResponse(data)) { ... throw err; }
}
// response.ok チェックが後に続くが、content-typeがJSONでfailedでない場合、
// レスポンスボディは既にconsumeされている可能性
```

`response.clone()` は成功時のみ有効。JSON解析失敗（`catch(() => null)`）の場合、ボディが部分消費されるリスク。

**修正**: non-streaming fallback 分岐を `response.ok` チェックの前に完全に分離。`response.clone()` は `response.ok` 確認後にのみ使用。

### H-3: `data-bff-token` 属性へのトークン注入（server.js:306-309）

```js
const injected = html.replace(
  /<body(\s*[^>]*)>/i,
  (match, attrs) => `<body${attrs} data-bff-token="${localAuthToken}">`,
);
```

`localAuthToken` は hex 文字列なので直接注入は安全だが、将来的にトークン形式が変わった場合のXSS脆弱性。CSPの `unsafe-inline` と組み合わせるとリスク増大。

**修正**: `localAuthToken` をHTMLエスケープするか、`data-bff-token` をCookie経由のみに変更（既にHttpOnly Cookieで送信されているため、data属性は不要冗長化の可能性）。

### H-4: agentセッションのcwd検証が不十分（agent.js:115）

```js
const validatedCwd = cwd ? path.resolve(cwd) : process.cwd();
```

セッション作成時に `cwd` を検証せず、任意ディレクトリを設定可能。`validatePath()` を呼んでいない。

**修正**: `cwd` が提供された場合、`validatePath(validatedCwd)` を実行。

### H-5: `app.use` のルート順序問題（server.js:319 vs 288-317）

```js
app.get(["/", "/index.html"], handler);  // Line 288
app.use(express.static(path.join(__dirname, "public")));  // Line 319
```

`express.static` は `GET /` も配信する可能性がある。`/index.html` のハンドラが `data-bff-token` を注入するのに、`express.static` が直接配信するとトークンなし。

**修正**: `express.static` の `index` オプションを `false` にするか、ルート順序を明示的に管理。

## 3. Medium（推奨修正）

### M-1: `firstTextCandidate` の結果が always fallback する問題（api-client.js:164-166）

```js
export function extractText(data) {
  return firstTextCandidate(data) ?? JSON.stringify(data, null, 2);
}
```

1min.aiの `resultObject` が空配列 `[]` の場合、`firstTextCandidate` は空文字列を返す（`Array.isArray` ブランチで `join`）。空文字列は truthy なのでフォールバックされない。

**修正**: `firstTextCandidate` の空配列/空文字列ケースを `undefined` に統一。

### M-2: CODE_GENERATOR の `conversationId` がリクエスト毎に生成される（web-search.js:66）

```js
conversationId: `CODE_GEN_${crypto.randomUUID()}`,
```

1min.ai APIドキュメントでは `conversationId` は必須。毎回UUID生成で会話履歴が蓄積されない。

**修正**: セッション単位で `conversationId` を保持するか、クライアントから受け取る。

### M-3: `sanitizeForPrompt` の制御文字除去が不完全（ai.js:478-483）

```js
.replace(/[\x00-\x1f\x7f]/g, "")
```

`\x09`(HT), `\x0a`(LF), `\x0d`(CR) も含まれるため、意図しない改行注入が可能。プロンプトインジェクションの可能性。

**修正**: `\x09`, `\x0a`, `\x0d` を許可するか、全て除去後に `trim()` する。

### M-4: `fs-guard.js` の `validatePath` でrealpathSyncが失敗した場合のフォールバック（fs-guard.js:93-109）

存在しないパスへの `realpathSync` 失敗時、既存の祖先を辿って合成パスを作成。この合成パスがallowed roots内にあると誤判定される可能性。

**修正**: 合成パスに対する追加のセキュリティチェック、または未存在パスは存在確認後にのみ検証。

### M-5: コマンド実行の `spawn` がシェルを通さない（command-runner.js:204-211）

Windows組み込みコマンド（`dir`, `echo`, `exit`）は `cmd.exe /c` で実行されるが、他のコマンドは `spawn(commandName, args)` で直接実行。これ自体は安全だが、`git` のような複雑なコマンドは引数展開が必要な場合がある。

**修正**: 現状は安全だが、将来的にシェル展開が必要なコマンドが追加された場合の考慮。

### M-6: `sanitizePayload` の再帰的走査に循環参照の防御なし（server.js:233-254）

```js
const sanitized = JSON.parse(JSON.stringify(payload));
```

循環参照がある場合 `JSON.stringify` が例外を投げ、`catch` で `"[Unable to sanitize details]"` にフォールバック。これは安全だが、エラーログの品質低下。

### M-7: agentセッションの `saveSessions()` が同期的IOでイベントループをブロック（agent.js:56-64）

大規模セッション時にファイル書き込みがイベントループをブロックする可能性。

**修正**: `writeFile` の代わりに `pipeline` またはバッファリング。

## 4. Low / Nit

### L-1: `parseCookies` の手動実装（server.js:64-76）

`cookie` パッケージを使うか、既存の `parseCookies` を信頼するか。現状はシンプルで問題なし。

### L-2: `compareAuthToken` のSHA-256ハッシュ（server.js:55-62）

元のトークンで `timingSafeEqual` を直接比較する方が高速。SHA-256は不要なオーバーヘッド。

### L-3: `cleanupExpiredSessions` のLRU削除が不完全（agent.js:127-132）

1000セッション上限到達時に最古を削除するが、`Array.from` で全エントリをコピーする。1000エントリでは問題なし。

### L-4: `web-search.js` の `buildWebSearchSettings` が未使用（web-search.js:42-48）

`buildWebSearchSettings` 関数が定義されているが、使用箇所がない。

### L-5: フロントエンドの `escapeHtml` が手動実装（app.js:1527-1535）

DOMPurifyが利用可能なら、`escapeHtml` もDOMPurifyに委任可能。ただし現在の実装も安全。

### L-6: `app.js` の2988行は単一ファイルとして過大

コーディング機能、チャット、画像、エージェントが全て1ファイル。保守性の観点から分割検討。

### L-7: `config/models.js` のモデル配列が `let` で再代入可能（models.js:6-127）

動的更新で上書きされるが、外部から変更される可能性。`const` + プロパティ更新が推奨。

## 5. 良い点

- **BFFパターンの徹底**: APIキーがフロントエンドに露出しない設計
- **二重認証**: `x-local-bff-token` ヘッダー + `__bff_session` HttpOnly Cookie
- **CSRF対策**: Origin/Host/sec-fetch-site/referer の複合検証
- **CSP設定**: Monaco CDN、Google Fontsを許可しつつframe/object を禁止
- **レート制限**: エンドポイント別に適切な値（autocomplete: 600, chat: 300）
- **Magic-byte MIME検証**: ファイルアップロードのMIME偽装防止
- **TOCTOU対策**: `revalidateRealPath()` で symlink swap 攻撃を軽減
- **コマンド実行の3層防御**: allowlist + shell metachar block + dangerous pattern
- **DOMPurify + marked**: XSS対策の正しい適用
- **retry + exponential backoff**: 429/5xxエラーへの耐性
- **クライアント切断時のAbortController**: リソースリーク防止
- **セッションTTL + 自動クリーンアップ**: メモリリーク防止
